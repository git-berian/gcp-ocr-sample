# AI駆動開発ガイドライン

AI（Claude）との協働における指示・品質・役割分担・セッション管理のルール。

---

## 1. 指示の原則

### 1指示 = 1目的

- 1つの指示で達成したいことを1つに絞る
- 複数の目的がある場合は分割して順番に依頼する

### 曖昧さの解消

- AIは曖昧な指示を受けたら、推測で進めずに質問して確認する
- 「よしなにやって」は禁止。具体的なゴールを明示する

### コンテキストの渡し方

指示時に以下を含めると精度が上がる：

- **Issue番号**: `#<番号>` で関連Issueを参照
- **関連ファイル**: 変更対象・参考にすべきファイルパス
- **背景・理由**: なぜその変更が必要か（What だけでなく Why）
- **制約**: パフォーマンス要件、互換性、期限など

---

## 2. AI生成コードの受け入れ基準

### テスト

- 実装とセットでユニットテストを生成する
- 既存のテストカバレッジを下げない
- テストが通ることを確認してから完了とする

### 型安全性

- `any` 型の使用を禁止する
- `strict: true` を維持する
- 型アサーション（`as`）は最小限にとどめる

### コード品質

- 既存のコードパターン・命名規則に従う
- デッドコード（未使用の変数・関数・import）を残さない
- 不要なコメントやTODOを残さない

### レビュー指摘への対処

- CodeRabbit等のレビュー指摘は原則すべて対処する
- 対処しない場合は、無視する理由をPRコメントに記載する

---

## 3. 役割分担

### AIが自律的に進めてよいこと

- バグ修正（ログ・エラー・テストから原因を特定して修正）
- リファクタリング（既存の動作を変えない範囲）
- テストの追加・修正
- ドキュメントの軽微な更新
- lint / format エラーの修正
- CIの失敗修正

### 人間の確認が必要なこと

- アーキテクチャの変更（レイヤー構成、ディレクトリ構造）
- 新しい依存パッケージの追加
- セキュリティに関わる変更（認証・認可・シークレット管理）
- 外部API連携の新規追加・変更
- データベーススキーマの変更
- 破壊的変更（既存APIのインターフェース変更）

### ゲートポイント

以下のアクションは人間が明示的に指示してから実行する：

- `git commit`（コミットメッセージの確認）
- `git push`
- PR作成
- Issue作成・クローズ

---

## 4. 品質管理

### セルフレビュー観点

コード変更時に以下の観点でセルフチェックを行う：

- **セキュリティ**: ユーザー入力のバリデーション、シークレットの露出がないか
- **エラーハンドリング**: 外部呼び出し・I/Oの失敗パスが適切に処理されているか
- **設計**: レイヤー違反がないか、既存パターンと一貫しているか
- **テスト**: 変更に対応するテストがあるか、エッジケースを考慮しているか

### コード受け入れ前チェックリスト

```
□ lint / format がパスする
□ 型エラーがない（tsc --noEmit）
□ テストが全件パスする
□ カバレッジが下がっていない
□ 不要な差分が含まれていない
□ コミットメッセージが規約に従っている
```

### パフォーマンス基準

- 不要なループ・再計算を避ける
- N+1クエリを作らない
- 大量データ処理にはストリーミングやバッチ処理を検討する

---

## 5. セッション管理

### 開始時

- `tasks/lessons.md` を読み込む（CLAUDE.md で必須化済み）
- 前回のセッションからの引き継ぎがあれば `tasks/todo-#<issue番号>.md` を確認する

### 中断時

- 作業中のタスクの進捗を `tasks/todo-#<issue番号>.md` に記録する
- 次回セッションで再開できるよう、残タスク・ブロッカー・方針を明記する

### 終了時チェックリスト

```
□ 作業ブランチの変更がコミット済み（またはスタッシュ済み）
□ テストが全件パスしている
□ todo ファイルに進捗が記録されている
□ lessons.md にセッション中の学びを追記している（該当する場合）
```
