# ADR-0001: クリーンアーキテクチャの採用

## ステータス

承認

## コンテキスト

Document AI を使った OCR 処理アプリケーションを構築するにあたり、外部サービス（Google Cloud Document AI）への依存をどう管理するかが課題だった。

## 決定

クリーンアーキテクチャに基づき、3層構造を採用する。

- **domain**: ビジネスルール。外部依存を持たない純粋な関数・型定義
- **application**: ユースケース。ドメイン層のインターフェースを通じてロジックを組み立てる
- **infrastructure**: 外部サービスとの接続。ドメイン層で定義されたインターフェースを実装する

## 理由

- 外部サービス（Document AI）の変更がドメインロジックに波及しない
- テスト容易性の向上（infrastructure をモックに差し替え可能）
- 責務の分離が明確になり、コードの見通しが良くなる

代替案として、レイヤー分離なしのフラットな構造も検討したが、Document AI のバージョンアップ（v8 → v9）時の影響範囲が大きくなるリスクがあった。

## 影響

- ファイル数は増えるが、各ファイルの責務が明確
- 新しい外部サービスを追加する際は infrastructure 層に実装を追加する
- レイヤー間の依存方向（domain ← application ← infrastructure）を守る必要がある
